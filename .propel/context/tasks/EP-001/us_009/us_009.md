# User Story - US_009

## Story ID

- ID Format: US_009

## Story Title

- Staff Patient Account & Auth Logging

## Description

- As a staff member, I want the ability to create patient accounts on behalf of walk-in patients and have all authentication events logged immutably, so that patients without email access can be onboarded and all access is auditable for HIPAA compliance.

## Acceptance Criteria

### AC-1: Staff can create patient accounts

- **Given**: A staff member is authenticated with Staff or higher role
- **When**: They submit the patient creation form with required fields (name, phone, DOB)
- **Then**: A patient account is created with status "active" (no email verification required), a temporary password is generated, and the creating staff member is recorded in the audit log

### AC-2: Temporary password with forced change

- **Given**: A staff-created patient account exists with a temporary password
- **When**: The patient logs in for the first time
- **Then**: They are forced to change their password before accessing any other feature, and the temporary password is invalidated

### AC-3: Authentication events logged immutably

- **Given**: Any authentication event occurs (login, logout, failed attempt, token refresh, password change)
- **When**: The event completes
- **Then**: An immutable audit record is created with (event_type, user_id, ip_address, user_agent, timestamp, success/failure, failure_reason if applicable)

### AC-4: Auth audit log queryable by admin

- **Given**: An admin user navigates to the auth audit section
- **When**: They search by user ID, date range, or event type
- **Then**: Results are returned paginated (50 per page) within 2 seconds, sorted by timestamp descending, with no ability to modify or delete records

## Edge Cases

- Staff creates account with phone number already in system → allow (patients may have multiple accounts via different staff); flag for potential merge review
- Audit log storage approaching capacity → trigger alert at 80% capacity; never drop audit records even under storage pressure
- Concurrent staff creating same patient → optimistic concurrency with retry; second creation fails with "patient recently created" message
