# User Story - US_008

## Story ID

- ID Format: US_008

## Story Title

- RBAC & Session Management

## Description

- As a system administrator, I want role-based access control with JWT session management, token refresh, and role hierarchy enforcement, so that each user type (patient, staff, provider, admin) accesses only authorized features.

## Acceptance Criteria

### AC-1: Role hierarchy with 4 tiers enforced

- **Given**: The system defines roles (Patient, Staff, Provider, Admin)
- **When**: A user authenticates and receives a JWT
- **Then**: The token contains role claims, and the API Gateway enforces access based on role: Admin > Provider > Staff > Patient, with each higher role inheriting lower-role permissions

### AC-2: JWT access token with short expiry

- **Given**: A user successfully authenticates
- **When**: A JWT access token is issued
- **Then**: The token has a 15-minute expiry, contains user ID, email, roles, and is signed with RS256 algorithm using a 2048-bit RSA key

### AC-3: Refresh token rotation implemented

- **Given**: A user's access token expires
- **When**: The client sends the refresh token to the refresh endpoint
- **Then**: A new access token and a new refresh token are issued, the old refresh token is invalidated (single-use), and the refresh token has a 7-day sliding expiry

### AC-4: Concurrent session limit enforced

- **Given**: A user is logged in on 3 devices (maximum allowed)
- **When**: They attempt to log in from a 4th device
- **Then**: The oldest session is terminated, its refresh token is revoked, and the new session is established with a notification sent to the user

### AC-5: Login endpoint with rate limiting

- **Given**: The login endpoint is publicly accessible
- **When**: More than 5 failed login attempts occur for the same account within 15 minutes
- **Then**: The account is temporarily locked for 30 minutes, a notification is sent to the account email, and subsequent login attempts return 429 with retry-after header

## Edge Cases

- Refresh token reuse detected (replay attack) → revoke ALL tokens for that user, force re-authentication, log security event
- Clock skew between services → 30-second leeway on token expiry validation
- Role changed while session active → existing token remains valid until expiry; new token reflects updated role on next refresh
