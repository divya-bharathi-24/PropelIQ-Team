# User Story - US_014

## Story ID

- ID Format: US_014

## Story Title

- Preferred Slot Swap

## Description

- As a patient, I want to request a swap to a preferred time slot when my desired slot is occupied, and be automatically notified when it becomes available, so that I can get my ideal appointment time without constant manual checking.

## Acceptance Criteria

### AC-1: Swap request submission

- **Given**: A patient has a confirmed appointment but wants a different time
- **When**: They select a currently-occupied preferred slot and click "Request Swap"
- **Then**: A swap request is created with status "pending", the patient is added to the waitlist for that slot, and a confirmation message shows their position in the queue

### AC-2: Automatic notification when slot becomes available

- **Given**: A swap request is pending for a specific slot
- **When**: That slot becomes available (original patient cancels or reschedules)
- **Then**: The first patient in the waitlist receives a push notification and email within 60 seconds, with a 15-minute window to accept the swap

### AC-3: Swap acceptance completes atomic exchange

- **Given**: A patient receives a swap availability notification
- **When**: They accept within the 15-minute window
- **Then**: Their existing appointment slot is released, the preferred slot is reserved for them atomically, both appointments are updated, and confirmation is sent to both parties

### AC-4: Swap expiry and queue advancement

- **Given**: A patient does not respond to the swap notification within 15 minutes
- **When**: The acceptance window expires
- **Then**: Their request is moved to "expired" status, the next patient in the waitlist is notified, and the slot remains available during transitions

### AC-5: Swap request limit per patient

- **Given**: A patient has active appointments
- **When**: They attempt to create more than 3 concurrent swap requests
- **Then**: The system prevents the 4th request with a message "Maximum swap requests reached. Complete or cancel existing requests first."

## Edge Cases

- Both patients in a direct swap cancel simultaneously → both slots released to general availability; no swap occurs
- Notification delivery fails (email bounce, push not registered) → retry once via alternative channel; if both fail, advance queue after 5 minutes (not 15)
- Swap requested for slot less than 2 hours away → reject with "Swap requests require at least 2 hours lead time"
