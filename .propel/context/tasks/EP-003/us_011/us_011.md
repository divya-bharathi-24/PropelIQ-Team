# User Story - US_011

## Story ID

- ID Format: US_011

## Story Title

- Real-Time Slot Display & Booking

## Description

- As a patient, I want to view real-time available appointment slots filtered by provider, specialty, and date, and book an appointment with instant confirmation, so that I can quickly find and secure a convenient appointment time.

## Acceptance Criteria

### AC-1: Slot availability displayed in real-time

- **Given**: A patient navigates to the booking page
- **When**: They select a provider or specialty and date range
- **Then**: Available time slots are displayed within 2 seconds, grouped by date, showing provider name, time, duration, and remaining capacity, refreshing every 30 seconds via WebSocket

### AC-2: Filter by provider, specialty, and date

- **Given**: The booking page is loaded
- **When**: A patient applies filters (provider name, specialty dropdown, date range picker)
- **Then**: Results update within 1 second, showing only matching available slots; empty results display "No available slots" with suggested alternative dates

### AC-3: Appointment booking with instant confirmation

- **Given**: A patient selects an available time slot
- **When**: They confirm the booking
- **Then**: The slot is reserved atomically (preventing double-booking), the appointment is created with status "confirmed", and a confirmation message is displayed within 3 seconds

### AC-4: Optimistic locking prevents double-booking

- **Given**: Two patients attempt to book the same slot simultaneously
- **When**: Both submit at the same time
- **Then**: Only the first request succeeds; the second receives a "slot no longer available" message within 2 seconds and the slot list refreshes automatically

### AC-5: Booking respects provider schedule constraints

- **Given**: A provider has defined working hours (e.g., 9AM-5PM, no Sundays)
- **When**: Slots are generated for display
- **Then**: Only slots within working hours appear, buffer time between appointments (10 min) is enforced, and max daily appointments per provider is not exceeded

## Edge Cases

- Redis cache stale when slot just booked by another user → WebSocket pushes invalidation event; UI shows "verifying availability" before confirming
- Patient books then immediately navigates away → booking persists; confirmation email sent regardless of browser state
- Provider schedule changed while patient is viewing slots → stale slot selection returns "schedule updated" error; force refresh of slot list
