# User Story - US_027

## Story ID

- ID Format: US_027

## Story Title

- Admin User Management

## Description

- As a system administrator, I want a comprehensive user management interface to create, modify, deactivate, and audit user accounts with role assignments, so that I can maintain platform access control and respond to staffing changes efficiently.

## Acceptance Criteria

### AC-1: User list with search, filter, and pagination

- **Given**: An admin navigates to user management
- **When**: The page loads
- **Then**: All platform users are displayed in a table with columns (name, email, role, status, last login, created date), searchable by name/email, filterable by role and status, paginated at 25 per page, with results loading within 2 seconds

### AC-2: Create new user with role assignment

- **Given**: An admin clicks "Create User"
- **When**: They fill in the user form (name, email, role, department)
- **Then**: The user account is created with the specified role, a welcome email with temporary password is sent via SendGrid, account status is set to "pending_activation", and the action is logged in the audit trail

### AC-3: Modify user role and permissions

- **Given**: An admin selects an existing user
- **When**: They change the user's role
- **Then**: The role change takes effect on the user's next token refresh (within 15 minutes), the change is logged with admin ID, previous role, new role, and reason (required field), and the affected user receives a notification

### AC-4: Deactivate user account

- **Given**: An admin needs to remove a user's access (e.g., employee termination)
- **When**: They click "Deactivate" and confirm with reason
- **Then**: The account status changes to "deactivated", all active sessions are terminated immediately (refresh tokens revoked), the user cannot login, but their data is retained per HIPAA requirements

### AC-5: Bulk user operations

- **Given**: An admin needs to perform actions on multiple users
- **When**: They select multiple users via checkboxes
- **Then**: Bulk actions available include: deactivate all, change role (same role for all selected), export to CSV, and send notification; each action requires confirmation and is individually audit-logged

## Edge Cases

- Admin attempts to deactivate their own account → reject with "Cannot deactivate your own account. Contact another administrator."
- Admin deactivates the last remaining admin → reject with "At least one active administrator must exist at all times"
- User email already exists (reactivation scenario) → present option to reactivate the existing deactivated account rather than creating duplicate
