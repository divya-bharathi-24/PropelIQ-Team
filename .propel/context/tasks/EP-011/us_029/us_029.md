# User Story - US_029

## Story ID

- ID Format: US_029

## Story Title

- Platform Reliability & Resilience

## Description

- As a platform engineer, I want circuit breakers, retry policies, graceful degradation, health checks, and performance monitoring configured across all services, so that the platform maintains high availability and recovers automatically from transient failures without manual intervention.

## Acceptance Criteria

### AC-1: Polly circuit breaker on all external dependencies

- **Given**: External services (Gemini API, SendGrid, external insurance APIs) are called
- **When**: 3 consecutive failures occur for a specific dependency
- **Then**: The Polly circuit breaker opens for 60 seconds, all subsequent calls to that dependency receive an immediate fallback response (cached data or graceful degradation message), and a recovery test is attempted after the break period

### AC-2: Retry policies with exponential backoff

- **Given**: A transient failure occurs (network timeout, 503, 429)
- **When**: The call fails
- **Then**: Polly retries up to 3 times with exponential backoff (1s, 2s, 4s) with jitter, logs each retry attempt, and if all retries fail, the circuit breaker failure count is incremented

### AC-3: Health check endpoints for all services

- **Given**: Each microservice is running
- **When**: The `/health` endpoint is called
- **Then**: It returns within 1 second with aggregate health status (Healthy/Degraded/Unhealthy), checking: database connectivity, Redis connectivity, dependent service availability, and memory/CPU thresholds (>90% = Degraded)

### AC-4: Graceful degradation when non-critical services fail

- **Given**: A non-critical service (Notification, ClinicalIntelligence) is unavailable
- **When**: A user performs an action that would call that service
- **Then**: The primary action succeeds (e.g., booking is saved), the secondary action is queued for retry (e.g., confirmation email), the user sees a non-blocking warning "Some features temporarily unavailable", and no data is lost

### AC-5: Performance monitoring and alerting

- **Given**: All services are instrumented with structured logging and metrics
- **When**: Key performance indicators degrade (p95 response time > 3s, error rate > 5%, queue depth > 100)
- **Then**: Alerts are triggered within 2 minutes via configured channels (email to ops team), dashboards display real-time metrics with 1-minute granularity, and auto-scaling triggers if supported by hosting tier

## Edge Cases

- All instances of a service crash simultaneously → YARP returns 503 for that service's routes with human-readable error; other services continue operating independently
- Circuit breaker in half-open state receives another failure → immediately re-open for double the previous break duration (120s), up to max 5 minutes
- Cascading failures (Service A depends on Service B, both circuit-break) → independent circuit breakers prevent cascade; Service A's fallback does not itself call Service B
