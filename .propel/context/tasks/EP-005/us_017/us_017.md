# User Story - US_017

## Story ID

- ID Format: US_017

## Story Title

- Manual Intake Form

## Description

- As a patient, I want a structured manual intake form as an alternative to the AI conversational intake, so that I can complete pre-appointment information in a traditional form format if I prefer or if the AI intake is unavailable.

## Acceptance Criteria

### AC-1: Multi-section intake form with validation

- **Given**: A patient chooses manual intake or is redirected from AI intake
- **When**: The form loads
- **Then**: It presents sections (Chief Complaint, Medical History, Current Medications, Allergies, Family History, Lifestyle) with required field indicators, input validation, and section-level progress tracking

### AC-2: Form supports save and resume

- **Given**: A patient is filling out the intake form
- **When**: They navigate away or close the browser
- **Then**: Entered data is auto-saved every 30 seconds to local storage and server-side draft, and they can resume from where they left off on any device

### AC-3: Conditional fields based on responses

- **Given**: A patient is completing the intake form
- **When**: They select a condition (e.g., "Currently pregnant: Yes")
- **Then**: Relevant follow-up fields appear dynamically (e.g., due date, prenatal care provider) without page reload, using Angular reactive forms

### AC-4: Submission creates same structured output as AI intake

- **Given**: A patient completes and submits the manual form
- **When**: The data is processed
- **Then**: It produces the same structured data format as the AI intake (symptoms, duration, severity, medications, allergies) so downstream provider views work identically regardless of intake method

## Edge Cases

- Patient switches from AI to manual mid-intake → pre-populate manual form with any data already captured in the AI conversation
- Form submission fails due to network error → retry with exponential backoff; show "submission pending" status; prevent duplicate submissions on reconnect
- Extremely long free-text entries (>5000 chars) → enforce character limit with counter; truncation warning at 4500 chars
